# Fact Check Skill

## Quick start

1. 把使用者輸入拆成可驗證主張，一次只查一個主張。
2. 先用 mcp_tavily_tavily_search 找 5-8 個候選來源。
3. 對核心來源用 mcp_fetch_fetch_readable 取正文；若抓不到內容，改用 mcp_fetch_fetch_markdown 或 mcp_fetch_fetch_html。
4. 至少比對 2 個獨立來源後再下結論。
5. 依固定格式輸出：結論、信心分數、來源清單、限制。

## Workflows

### 1) URL 連結查核

- [ ] 抽取 URL 內文的主要主張（人、事、時、地、數字）。
- [ ] 搜尋同事件的其他來源（官方、主流媒體、原始資料）。
- [ ] 若來源網站反爬蟲，先保留原 URL，再用 fetch 與鏡像源補充證據。
- [ ] 標記每個來源的發布日期與是否二手轉述。
- [ ] 若敘述涉及統計數字，優先找原始報告或官方資料庫。
- [ ] 輸出查核結論，並附可點擊來源。

### 1.1) 反爬蟲 fallback（Tavily + Fetch + 鏡像）

- [ ] 第 1 層：mcp_tavily_tavily_search 找候選來源與同事件報導。
- [ ] 第 2 層：用 mcp_fetch_fetch_readable 擷取原文；失敗時改 mcp_fetch_fetch_markdown / mcp_fetch_fetch_html / mcp_fetch_fetch_txt。
- [ ] 第 3 層：若原站封鎖，使用鏡像源抓取同內容，再回查原始事件脈絡。
- [ ] x.com 案例：可嘗試 x.com 原文 + 鏡像（例如 fxtwitter、vxtwitter、nitter instance）比對時間與文本。
- [ ] 鏡像內容僅作輔助證據，最終仍要回到一手來源或多家獨立報導交叉驗證。

### 2) 純文本查核

- [ ] 先改寫主張為「可被證偽」的句子。
- [ ] 拆分子主張（時間、地點、人物、數據、因果）。
- [ ] 逐項搜尋並做證據矩陣（支持/反駁/未證實）。
- [ ] 若互相矛盾，提升不確定性並解釋原因。
- [ ] 若證據不足，結論必須是「證據不足」，不可硬判真偽。

### 3) 圖片查核

- [ ] 先辨識圖片中的可查元素：地標、文字、車牌、制服、天氣、陰影。
- [ ] 把可查元素轉成關鍵字做網路搜尋。
- [ ] 搜尋是否有相同場景於不同時間或地點被重用。
- [ ] 若無法取得可靠反向查圖結果，明確列為限制。
- [ ] 僅在有可追溯證據時下最終結論。

## Evidence rules

- 來源優先順序：官方資料 > 一手報導 > 專業機構 > 一般媒體 > 社群貼文。
- 避免只依賴單一來源，至少要有兩個互相獨立的來源。
- 鏡像源可用於取得內容，但不可單獨作為最終定論依據。
- 優先使用最近且可追溯資料，注意時區與更新時間。
- 引用時要附標題與 URL，必要時附關鍵摘錄。

## Output format

請固定輸出以下欄位：

- 結論: 真 / 假 / 部分正確 / 證據不足
- 信心分數: 0-100
- 核心理由: 2-4 點
- 來源清單: 至少 2 筆（標題 + URL）
- 限制與不確定性: 1-3 點

## Failure handling

- 找不到足夠證據時，不推測、不補空白，直接標示「證據不足」。
- 若主張範圍過大，主動要求縮小問題範圍（事件、時間、地區）。
- 若圖片資訊不足，要求更高解析度或原始上下文。

## Advanced features

- 針對爭議主題可使用 mcp_tavily_tavily_research 做更深度彙整。
- 需要快速初篩時，先用 mcp_tavily_tavily_search（fast 或 advanced）再深入。
- 遇到反爬蟲或內容擋牆時，優先啟動 fetch 多格式擷取，再用鏡像源補證。
